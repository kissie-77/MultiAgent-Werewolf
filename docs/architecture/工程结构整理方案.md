# 工程结构整理方案

## 目标

本方案用于在不推翻现有六大板块的前提下，逐步整理文件归属和依赖方向。第一阶段先治理依赖边界，不急于搬文件，避免把循环依赖和职责混乱一起迁移到新目录。

六大板块固定为：

| 板块           | 职责                                                                    |
| -------------- | ----------------------------------------------------------------------- |
| `game_runtime` | 狼人杀规则、角色、动作、状态、事件、引擎流程、配置、序列化、胜负判断    |
| `agent_team`   | Agent 执行、AgentScope 接入、消息路由、信息隔离、记忆、skill 读取       |
| `strategy`     | Prompt、结构化决策 schema、阶段输出契约、投票意向模型                   |
| `evaluation`   | 复盘、评分、日志视图、Coach、skill 生成、prompt proposal、bad case 分析 |
| `interface`    | CLI/TUI/eval 入口、配置加载、模式选择、跨板块装配                       |
| `ui`           | 展示组件和 presenter，只读事件与状态                                    |

## Coach 与 Skill 定位

Coach 属于 `evaluation`，当前路径为：

```text
src/llm_werewolf/evaluation/post_game/coach/
```

Coach 负责赛后复盘、经验提炼、skill 生成和 skill 权重建议。`agent_team` 只负责读取 skill、注入记忆、辅助 Agent 决策，不负责生成 skill。

Skill 生命周期固定为单向链：

```text
game_runtime 产生日志/事件
evaluation/post_game/coach 复盘并生成 skill
evaluation 写入 agent_team/skills
agent_team/skill_loader 读取 skill
agent_team/memory 注入描述或上下文
Agent 使用 skill 决策
```

关键规则：

- `evaluation` 可以写入 `agent_team/skills`。
- `agent_team` 只读取 `agent_team/skills`。
- `game_runtime` 不读写 skill。
- `strategy` 不读写 skill。
- `ui` 不读写 skill。

## 允许依赖方向

`interface` 是装配层，可以依赖其他板块：

```text
interface -> game_runtime
interface -> agent_team
interface -> evaluation
interface -> ui
interface -> strategy
```

`evaluation` 是赛后分析层，可以消费运行结果和策略契约：

```text
evaluation -> game_runtime
evaluation -> strategy
evaluation -> agent_team
```

`agent_team` 是 Agent 执行层，可以依赖运行时公共协议和策略契约：

```text
agent_team -> game_runtime.types
agent_team -> game_runtime.config
agent_team -> game_runtime.prompts
agent_team -> strategy
```

`ui` 是展示层，只允许读取运行时状态和事件：

```text
ui -> game_runtime
ui -> ui
```

`game_runtime` 是规则和状态核心层，长期目标是不依赖 Agent 实现：

```text
game_runtime -> game_runtime
game_runtime -> strategy 的稳定 DTO / 决策契约
```

## 禁止依赖方向

以下方向默认禁止：

```text
game_runtime -> agent_team
game_runtime -> evaluation
game_runtime -> interface
game_runtime -> ui

strategy -> agent_team
strategy -> evaluation
strategy -> interface
strategy -> ui

agent_team -> evaluation
agent_team -> interface
agent_team -> ui

ui -> agent_team
ui -> evaluation
ui -> interface
```

当前这轮整理后，`game_runtime -> agent_team` 的已知历史债务已经清零。后续继续通过架构测试防止回流。

## 当前已知债务

运行时交互 Hub 已改为由 `interface/bootstrap.py` 创建并注入 `GameEngine`，`game_runtime` 不再直接依赖 `agent_team` 实现。

## 队友快速放置指南

新文件如果和“游戏规则怎么跑”有关，放 `game_runtime`：

- 角色规则、阶段流程、状态、事件、胜负判断
- 不要在这里直接 import `agent_team`

新文件如果和“Agent 怎么想、怎么说、怎么接 LLM”有关，放 `agent_team`：

- Agent 实现放 `agent_team/agents/`
- 通信与 MsgHub 路由放 `agent_team/communication/`
- 调用封装放 `agent_team/invocation/`
- 记忆放 `agent_team/memory/`
- skill 读取和 markdown 解析放 `agent_team/skill_support/`

新文件如果和“策略契约/Prompt 模板/结构化输出”有关，放 `strategy`：

- schema、phase output、投票意向模型、Prompt 模板

新文件如果和“赛后复盘、评分、日志视图、skill 生成、Coach”有关，放 `evaluation`：

- Coach 放 `evaluation/post_game/coach/`
- skill 生成放 `evaluation/post_game/skill_generation/`

新文件如果只是“把各板块组装起来”，放 `interface`：

- CLI、TUI、eval runner 的装配入口
- 具体实现对象的创建，例如 `InformationHub()`

新文件如果只是“展示”，放 `ui`：

- presenter、显示组件、界面层格式化

## 一句话原则

`game_runtime` 负责对局发生，`agent_team` 负责玩家怎么响应，`strategy` 负责响应格式和策略表达，`evaluation/Coach` 负责赛后学习和生成 skill，`interface` 负责装配，`ui` 负责展示。
