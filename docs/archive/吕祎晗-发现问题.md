## 问题 6：事件日志双轨（EventLogger vs GameState.event_history）

- **文件**：
    - `src/llm_werewolf/game_runtime/events.py`（`EventLogger`）
    - `src/llm_werewolf/game_runtime/game_state.py`（`event_history` / `record_event`）
    - `src/llm_werewolf/game_runtime/engine/base.py`（引擎 `_log_event` → `EventLogger`）
- **现状**：
    | 路径                      | 写入方                              | 读取方                                   | 是否活跃    |
    | ------------------------- | ----------------------------------- | ---------------------------------------- | ----------- |
    | `EventLogger.events`      | 引擎 `_log_event`、各 phase handler | 玩家 observation 构建、evaluation runner | ✅ 活跃     |
    | `GameState.event_history` | `record_event()`                    | `get_recent_events()`                    | ❌ 无调用方 |
- **可行性分析**（保留 `event_history` 供后续日志 / 存档）：
    1. **短期（低改动）**：在 `GameEngine._log_event` 创建事件后追加 `game_state.record_event(event)`，双写保持同步；evaluation 仍读 `EventLogger`
    2. **中期（存档）**：在 `GameStateSnapshot` 增加可选 `events: list[EventSnapshot]`，save/load 时序列化 `EventLogger.events` 或 `event_history`（需 JSON 化 `Event`，`phase`/`event_type` 用 `.value`）
    3. **长期（统一）**：以 `GameState` 为唯一事件源，`EventLogger` 退化为薄封装或删除；需改 engine / evaluation / UI 全部读路径
    4. **注意**：`Event.visible_to` 支持信息隔离，存档时需完整保留；事件量随局数线性增长，全量存档可接受，回放可按 `visible_to` 过滤
- **建议**：先采用方案 1 双写验证，再决定是否把 events 纳入 snapshot；**本轮不改动运行时行为**，仅记录方案
- **状态**：📋 已分析，待后续实现
