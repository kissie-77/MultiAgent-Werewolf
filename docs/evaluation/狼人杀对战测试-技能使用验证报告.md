## 五、发现与建议

### ⚠️ 发现 1：deepseek-v4-flash 的结构化决策被丢弃（重要 · 已用受控复现证实）

> 注：先前版本把根因推断为「token 耗尽 + `except: pass` 静默回落」，经受控复现**该推断不成立**，已修正如下。

**受控复现**（`_play/repro_witch.py`：直接对真实模型调用 `WerewolfAdapterBridge.request_witch_night_choice`，首夜、有解药、刀口=3号）：

| 模型 | invoke_structured 返回 | `generate_response` 工具调用 | 最终决策（n 次） |
|---|---|---|---|
| deepseek-v4-flash | `None` ×5 | **0 次**（5 次全把决策写进 content 文本，如 `{"action":"save","seat":0,…}`） | **save=0 / none=1 / poison=4** |
| doubao-seed-2.0-pro | `WitchNightDecision(save)` ×3 | 3 次（`tool_use: generate_response`） | **save=3** |

**真正根因（两个缺陷叠加）**：
1. **B1 — deepseek 不发工具调用**：`deepseek-v4-flash` 把结构化决策当作**纯文本 JSON** 写进 `content`（它其实**正确地选了 save**！），而**不**调用 AgentScope `ReActAgent` 依赖的 `generate_response` 工具。于是 `get_structured_response` 取不到 `Msg.metadata` → 返回 `None`，且**不抛异常**（`structured_invoke.py` 仅在异常时 `log.warning`，返回 None 时无任何日志 —— 这才是“静默”的真正出处；复现中 deepseek 的 structured 警告数 = 0）。doubao 会真实发出 `tool_use`，故正常。
2. **B2 — 夜间/投票的文本兜底是坏的**：`invoke_structured` 返回 None 后，`bridge.py` 的 `request_witch_night_choice` / `request_seat_choice` 回落到 `agent.get_response`；但女巫/投票 prompt 被 `_message_expects_seat_only` 判为“仅座位号”，`get_response` 返回的**不是模型真实回答**，而是一段**写死的角色套话**（`agentscope_agent.py:_generate_role_specific_fallback`，如“我手里有药，但不会轻易使用……”）。bridge 再用 救/毒 关键字匹配这段套话 → 没“救”就变 `none`，含“毒”就变 `poison seat=0`（无效空毒）。**模型本来已在 content 里给出的 save，全程无人解析。**

**影响（已证实）**：deepseek 女巫 `save` **5/5 全部丢失**（0 救），正好解释 pure_ds 的「0 个 `💊` + 女巫困惑 + 投票仅 2/5 生效」；与「token / max_tokens」无关（复现里 deepseek 输出短且完整）。deepseek 的**发言**没事——发言 prompt 不属于“仅座位号”，文本兜底会返回模型真实发言文本并被 `parse_speech` 正确解析。doubao 全程不受影响。

**修复（更精准，按收益排序）**：
1. **在 `get_structured_response` 里解析 content-JSON**：当 `metadata` 为空但 `text` 是合法 JSON 时，先 `model_validate_json`/正则抽取再返回 —— **一处改动即可救回 deepseek 的真实决策**（它们本就是合法 JSON），模型无关、收益最高。
2. **结构化决策的兜底不要走 `get_response` 套话路径**：WitchNight / SeatChoice 在无结构化结果时应解析**原始模型文本**，而非写死的角色套话。
3. 给 deepseek 开启 `tool_choice` / function-calling 或 JSON mode；并在 `get_structured_response` 返回 `None` 时补 `log.warning`，避免“静默丢决策”。

**2026-05-28 复测结论（已修复）**：

- 真实 API 暴露出第二层问题：`deepseek-v4-flash` thinking mode 会拒绝强制 `tool_choice`，返回 `Thinking mode does not support this tool_choice`。修复后 `get_structured_response` 在该错误下改走普通模型调用，再从 content 中恢复结构化决策。
- 女巫最小请求：模型正文输出 `{"action":"save","seat":0,...}`，系统成功恢复为 `WitchNightDecision(save)`。
- 座位选择最小请求：模型正文输出 `{"seat":5,...}`，系统成功恢复为 `SeatChoiceDecision(seat=5)`。
- 真实 6 人 DeepSeek 对局产物：`runs/doubao-9p-20260528-005938`，共 38 条事件，正常结束，无 `Traceback` / `ERROR` / `BadRequest` / `tool_choice` 写入日志。
- 回归测试：`uv run --group test pytest -o addopts='' -q` 通过，结果 `351 passed`。

### ⚠️ 发现 2：doubao-seed-2.0-pro 极度啰嗦 → 速度很慢

每个决策产出海量 reasoning，并叠加额外的「Now generate a text response」ReAct 文本回合；单是 day-1 讨论就跑了约 20 分钟 / 3000+ 行。`vote-intention` 在每条发言后对全体采集，进一步放大开销。

**建议**：关闭或抽样 vote-intention 采集；限制 ReAct 文本回合数；或对决策类调用使用更短的提示。

### 发现 3（轻微）：女巫夜 2 提示自相矛盾

提示同时出现 `今晚狼人目标是玩家4` 与 `解药已耗尽，本夜不会告知刀口身份` —— 解药用完后仍透露了刀口（疑似观测上下文未做信息隔离）。影响小，建议核查 `_merge_private_context` / 观测构建对女巫的字段。

## 六、人类玩家体验反馈

- **提示清晰**：每次轮到我，系统给出精简提示 + 输入格式（`救/毒 座位号/0`、座位号、≥15 字发言），面向 LLM 的 schema 噪声已被过滤。

- **发言真能左右牌局**：两局都靠“起跳神职 + 带队投票”赢下；doubao 局里我（真女巫）甚至被狼人“悍跳女巫”对刚，最终靠预言家银水背书 + 毒杀收尾。整体体验接近真人局。
