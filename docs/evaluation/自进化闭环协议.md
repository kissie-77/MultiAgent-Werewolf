# 自进化闭环协议

本文档固定自进化 runner 的工程口径，避免后续在“什么算一版 Agent”“下一轮继承什么”“怎么证明提升”上反复改目标。

## 1. 一版 Agent 的进化对象

一版 Agent 由以下四类变量共同定义：

1. `prompt_version`
   - 当前运行时实际加载的 Prompt 版本。
   - 来源是 `strategy/prompts/<version>/` 或 evolution 生成的 `prompt_versions/<version>/`。
   - 赛后 `prompt_proposals.json` 被采纳后，会生成下一版 `prompt_version`。

2. 当前运行时生效的 `active skill` 集
   - 只包含 `status == active` 的 skill。
   - 下一轮开始前，从上一轮 `version_manifest.json.active_skills` 恢复到运行时 skill 库。
   - `draft` 不默认进入运行时，`deprecated` 永远不进入运行时。

3. 必要的 memory / 注入参数
   - 版本摘要必须记录影响运行时行为的 memory 参数。
   - 当前字段为 `memory_runtime_params`：
     - `skill_top_k`
     - `semantic_top_k`
     - `similarity_threshold`
     - `working_compression_enabled`

4. 模型参数
   - 模型参数算版本变量。
   - 当前必须记录：
     - `model`
     - `reasoning_effort`
   - 原因：模型能力和推理强度会直接影响胜率、发言质量和策略执行，不能和 Prompt / Skill 的效果混在一起比较。

版本摘要必须至少写入：

- `version_id`
- `prompt_version`
- `active_skills`
- `memory_runtime_params`
- `model_config`
- `prompt_evolution`

## 2. 每轮执行协议

`run_evolution_cycle()` 的标准轮次顺序固定为：

1. 计算本轮 `version_id`
   - 第 1 轮：`v1_initial`
   - 第 N 轮：`vN_evolved`

2. 继承上一轮 active skill 集
   - 第 1 轮不恢复。
   - 第 2 轮起，读取上一轮 `version_manifest.json.active_skills`。
   - 清空当前运行时 skill markdown。
   - 将上一轮 active skill markdown 回填到运行时 skill 库。

3. 使用当前 `prompt_version` 跑本轮评测
   - `EvaluationRunner.prompt_version` 必须等于本轮实际运行时版本。
   - PostGame pipeline 必须使用同一个 `prompt_version` 生成 proposal。

4. 赛后生成 Prompt 下一版
   - 读取本轮 `prompt_proposals.json`。
   - 采纳符合规则的 proposal。
   - 生成下一轮 `prompt_version`。
   - 产出：
     - `applied_prompt_proposals.json`
     - `prompt_version_diff.json`
     - `new_prompt_version.txt`

5. 写本轮 `version_manifest.json`
   - 记录本轮实际使用的 `prompt_version`。
   - 记录本轮赛后生成的 `prompt_evolution`。
   - 记录当前 active skill 快照。

6. 更新下一轮输入
   - `previous_run_dir = 本轮 run_dir`
   - `previous_version_manifest_path = 本轮 version_manifest.json`
   - `current_prompt_version = prompt_evolution.new_prompt_version`

7. 全部轮次结束后写全局结果
   - `leaderboards/leaderboard.json`
   - `ab_reports/*`
   - `evolution_summary.json`

## 3. 每轮目录结构

每轮目录固定为：

```text
<output_root>/
  v1_initial/
    games/
    leaderboard_entry.json
    experiment_meta.json
    prompt_proposals.json 或 games/*/prompt_proposals.json
    applied_prompt_proposals.json
    prompt_version_diff.json
    new_prompt_version.txt
    version_manifest.json
  v2_evolved/
    ...
  prompt_versions/
    <generated_prompt_version>/
      manifest.yaml
      variables.yaml
      roles/
      text/
  leaderboards/
    leaderboard.json
  ab_reports/
  evolution_summary.json
```

说明：如果 proposal 只在单局 `games/<game_id>/` 下生成，runner 后续可以继续增强为跨 game 聚合；当前最小闭环先按 run 级产物消费。

## 4. 继承规则

下一轮继承上一轮的规则固定为：

- Prompt：继承上一轮赛后 `prompt_evolution.new_prompt_version`。
- Skill：继承上一轮 `version_manifest.json.active_skills`。
- Memory 参数：沿用 runner 配置或默认 `_default_memory_runtime_params()`。
- Model 参数：沿用 runner 入参，变更模型必须形成新的实验组。

不继承：

- `draft skill`
- `deprecated skill`
- 未采纳的 prompt proposal
- 单局临时工作记忆
- 单局 episode 原文

## 5. 自进化验收口径

默认验收实验：

- scenario：`smoke_6p_basic`
- rounds：至少 2 轮，建议 3 轮
- games_per_round：开发阶段 1-3 局，汇报阶段至少 10 局，正式对比建议 20 局
- seed：固定初始 seed，并按轮次递增
- 对比对象：`v1_initial` vs 最后一轮

“提升成立”的工程版标准：

- 完成率不下降。
- 信息隔离、角色技能、阶段顺序等硬约束无新增严重失败。
- 终局版本相对初始版本在 leaderboard 核心指标上更好，优先看：
  - `win_rate`
  - `avg_mvp_score`
  - `avg_benefit_score`
  - `avg_intention_score`
- `evolution_summary.json` 能证明：
  - Prompt 版本发生过继承或明确记录未变化原因。
  - active skill 集发生过继承。
  - 初始版和终局版存在 A/B 报告。

如果做答辩展示，建议口径为：

```text
固定 scenario 和模型，连续跑 N 轮；
每轮赛后自动提炼 Prompt / Skill；
下一轮自动继承上一轮生效版本；
最终用 leaderboard 和 A/B 报告比较 v1_initial 与终局版本。
```

## 6. 后续统计增强范围

当前阶段从工程版趋势升级为基础统计版 A/B。

已纳入当前阶段：

- 多轮 win_rate / completion_rate 趋势
- active skill 数量变化
- prompt version chain
- 初始版 vs 终局版 A/B
- 版本 manifest 可回溯
- A/B 报告中的胜率 95% Wilson 置信区间
- A/B 报告中的双样本比例 z 检验 p-value
- `win_rate_significant` 显著性标记
- 多 scenario / 多 seed 实验矩阵
- `evolution_matrix_summary.json`
- `evolution_matrix_summary.md`

下一阶段建议增强：

- 增加“Prompt 变化”和“Skill 变化”的贡献拆分实验
- 小样本时补 Fisher 精确检验
- 矩阵实验的自动失败重试和预算控制

暂不做：

- 自动回滚主策略切换
- 大模型裁判自动决定 proposal 是否采纳
- 多模型混跑下的统一胜率显著性结论

原因：这些能力会扩大变量数量，容易让当前闭环验收口径变模糊。先把“能自动进化、能继承、能比较、能回溯”的主链路打稳。
