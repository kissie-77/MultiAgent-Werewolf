# LLM Compressor 重试与降级修复说明

## 背景

`agent_team/memory/llm_compressor.py` 负责在每轮结束时，把工作记忆中的动态信息压缩成简短摘要，供后续决策阶段参考。

这条链路依赖外部 OpenAI 兼容接口。一旦接口不可用，系统会退回到本地 fallback 压缩，保证游戏流程不停。

这次修改的目标不是改变记忆架构，而是增强这条依赖链的容错能力和日志可观测性。

## 改前

改动前的行为：

1. 每次压缩只请求一次外部 LLM。
2. 只要这一次失败，就立刻走 fallback。
3. 失败日志只会打一次。
4. 后续持续失败时，系统仍然继续 fallback，但基本不再提醒。

这会带来两个明显问题：

1. 瞬时故障无法自恢复  
   例如超时、网络抖动、临时 5xx，这类问题本来可能第二次请求就恢复，但原实现不会重试。

2. 长期故障难以感知  
   原实现只有第一次失败会打 warning，后续几十次失败也不会继续提示，容易让人误以为系统正常，只是摘要质量下降。

## 改后

这次修改后，`LLMCompressor` 的行为变成：

1. 外部压缩调用失败时，最多重试 3 次。
2. 重试采用指数退避，间隔为 `0.5s -> 1.0s -> 2.0s`。
3. 所有重试都失败后，再退回 fallback 压缩。
4. 连续失败时不再永久静默，而是：
   - 第 1 次失败会打 warning
   - 此后每累计 5 次失败再打一条 warning
5. 一旦后续某次压缩恢复成功，失败计数会清零。

## 为什么这样改

这样改主要是出于三个考虑：

1. 保持游戏流程稳定  
   fallback 仍然保留，外部服务挂掉时游戏不会中断。

2. 提高瞬时故障恢复率  
   很多失败并不是永久性故障，有限重试能明显提升成功率。

3. 提高问题可见性  
   既避免每次失败都刷屏，也避免长期静默，让开发和排查时能及时知道压缩链路是否持续异常。

## 影响范围

这次修改只影响：

- `src/llm_werewolf/agent_team/memory/llm_compressor.py`

不会改变：

- working memory 的整体结构
- semantic memory 的读写方式
- episodic memory 的职责
- 决策 prompt 的注入策略

## 当前结论

这次修复属于“依赖链容错增强”，不是记忆架构重构。

修完后系统在外部压缩服务异常时会更稳，也更容易发现问题，但不会改变你现在记忆板块的职责划分。
