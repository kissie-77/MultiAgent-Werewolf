## 总体目标

按已确认的工程质量方案，修复会影响对局正确性、信息隔离、复盘判断和长期维护的问题。

执行节奏：

## 关键决策

- smoke 对局优先使用 demo/offline 配置，避免外部模型服务波动干扰代码验证；用户指定时可用正式对局替代检查点。
- P0 优先修复会影响对局结果或 import 崩溃的问题。
- 大文件拆分放到后段，避免在关键行为修复前引入过大结构变动。
- 女巫毒药动作事件采用新名称 `witch_poison_used`；历史 artifacts 中的 `witch_poisoned` 保留兼容读取，避免旧复盘失效。
- 全仓 `ruff check src tests` 当前仍会被历史债打爆，因此 P1 不把全量 ruff 作为硬门禁；先用 `E9,F821` 这类运行级错误做渐进拦截，后续再分批提升规则范围。
- 子代理最多同时保留两个；本轮 CI 子代理因 429 失败，相关 CI 收口已由主线本地完成。
- P2 当前不直接推进到 75%，因为 75% 检查点仍需用户用正式对局或指定 smoke 对局确认流程可跑通。

## 后续注意事项

- 每次修改事件类型或事件字段，都需要同步可见性、复盘、checker 和测试。
- 每次调整 fallback，都需要检查是否可能改变胜负结果。
- 不纳入多人类玩家专项重构，当前项目目标仍按单人类混战处理。
- P1 的 50% 检查点已用正式对局替代 smoke 完成；后续 75%/100% 检查点仍按用户指定选择 smoke 或正式对局。
- 正式对局中仍可能出现外部模型超时或 AgentScope 中断诊断；可恢复诊断已降噪，后续若仍频繁失败再单独收口调用链超时策略。
