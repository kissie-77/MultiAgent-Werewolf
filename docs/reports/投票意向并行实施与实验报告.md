# 投票意向并行实施与实验报告

> 日期：2026-05-28
> 分支：`codex/vote-intention-parallelism`
> 目标：把“每次发言后全员投票意向采集”从完全串行改为受控并行，并量化对墙钟时间与对局记录的影响。

## 1. 实施摘要

本次没有全局拆除 AgentScope 串行锁，而是新增了一个**上下文局部绕锁**：

- 默认路径仍走 `run_serial_agent_call()` 的全局锁，保护发言、夜晚技能、赛后 replay 等高风险路径。
- 只有 `_collect_vote_intentions()` 在 `vote_intention_concurrency > 1` 时，才在当前 async context 内允许并行 Agent 调用。
- 外层使用 `asyncio.Semaphore` 限制 fan-out 宽度，避免把所有 provider 压力一次性打满。
- 并行采集使用 `asyncio.gather(..., return_exceptions=True)`，单个玩家的意向请求失败时只记录 warning 并跳过，不中止整轮讨论。

关键文件：

- `src/llm_werewolf/agent_team/serial_calls.py`

## 2. 风险控制

GitNexus 影响分析结果：

| 符号                       | 风险     | 结论                                                   |
| -------------------------- | -------- | ------------------------------------------------------ |
| `_collect_vote_intentions` | LOW      | 只被 `run_roundtable` 直接调用，适合局部改造           |
| `run_serial_agent_call`    | CRITICAL | 影响 Agent 调用、结构化调用、赛后 replay，不能粗暴拆锁 |
| `_wire_game_state`         | CRITICAL | 影响 setup/load/eval/CLI/TUI，改动仅限配置注入         |

因此本次采用“默认串行 + 局部 opt-out”的实现，而不是把锁替换成全局并发。

## 3. 时间计算模块

新增 `llm_werewolf.evaluation.time_analysis`，用于计算理论墙钟收益：

```text
串行：N 次发言 + (N + 1) 轮全员意向，每轮 N 个 call
并行：N 次发言 + (N + 1) 轮全员意向，每轮 ceil(N / concurrency) 个 call-time
```

6 人局，`concurrency=1 -> 6`，假设每个逻辑决策约 2 次 HTTP、每次 HTTP 10s：

| 项目             | 实施前 | 实施后 |
| ---------------- | -----: | -----: |
| 单轮讨论逻辑调用 |     48 |     48 |
| 估算墙钟         |   960s |   260s |
| 节省             |      - |   700s |
| 加速比           |      - |  3.69x |
| 节省比例         |      - |  72.9% |

注意：并行不减少 API 请求数，只减少同一批意向请求的等待时间。

## 7. 结论

本次改造达成目标：

- 投票意向 fan-out 已支持受控并行。
- 默认行为仍是串行，避免对高风险 Agent 调用路径造成全局影响。
- 6 人局理论单轮讨论墙钟从约 960s 降到约 260s，估算节省 72.9%。
- 实验记录显示并行没有改变记录中的意向内容；完整 Demo 对局也没有改变结构化结果。
- 真实 DeepSeek 完整对局实测：总耗时从 964.3s 降到 229.6s，提升 4.20x；投票意向环节从 446.8s 降到 45.4s，提升 9.84x。
- 真实 Doubao 放宽超时后串行/并行均完整跑完：总局耗时从 1214.6s 到 1194.3s，仅小幅节省 1.7%；但可比意向批次从 420.4s 到 119.3s，提升 3.52x，说明瓶颈确实被压低，只是总局被更多发言/意向事件抵消。
- Doubao 限流探针与未完成 c6 对局样本均未观察到 429、quota 或 rate-limit；当前证据支持“c6 并发可运行”，但更高并发或更大样本仍需单独压测。
- Doubao 12 人 / c12 探针同样未观察到 429、quota 或 timeout；不过 c12 的 p95 延迟从 c6 的 4.897s 升到 10.176s，吞吐没有继续增加，因此 c12 可用但不一定是最佳并发点。
