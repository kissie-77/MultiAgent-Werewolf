# 狼人杀 · 决策可靠性问题与修复方案

> 配套验证报告：`docs/evaluation/狼人杀对战测试-技能使用验证报告.md`
> 复现脚本：`_play/repro_witch.py`
> 日期：2026-05-26 ｜ 更新：2026-05-28 ｜ 状态：P1 已实施并真实 DeepSeek 复测；P2/P3 待实施

本文汇总人机 & 纯 Agent 对战测试中暴露的问题，并给出修复设计。问题按严重度排序：**P1（严重 · 决策丢失）> P2（性能 · 对局过慢）> P3（轻微 · 信息泄露）**。

**2026-05-28 实施状态**：本次分支 `codex/deepseek-structured-recovery` 已完成 P1 修复：

- `structured_invoke.parse_structured_from_text`：从模型正文中恢复结构化 JSON，兼容 markdown 围栏、前后散文、`arguments/input/structured_output` 包装，以及旧式 `[[seat]]` / 裸数字 / YES / NO 标量输出。
- `AgentScopeWerewolfAgent.get_structured_response`：当 `Msg.metadata` 为空时，非 Speech schema 不再直接返回 `None`，而是尝试解析 content；若 DeepSeek thinking mode 拒绝强制 `tool_choice`，会改走普通模型调用并再次解析 content。
- `generate_response_instruction(schema_name)`：非 Speech schema 不再复用 `SpeechDecision Schema` / `public_speech / private_thought` 指令，避免女巫、座位选择、投票意向 prompt 被发言 schema 污染。
- 可观测性：`invoke_structured` 最终返回 `None` 会记录 `structured_invoke_returned_none`；content 解析失败会记录 `structured_text_recovery_failed`。

真实验证：

- 真实 `deepseek-v4-flash` 女巫最小请求：content JSON 恢复为 `{"action":"save","seat":0,...}`，不再走旧文字兜底。
- 真实 `deepseek-v4-flash` 座位选择最小请求：content JSON 恢复为 `{"seat":5,...}`。
- 真实 6 人 DeepSeek 对局：`runs/doubao-9p-20260528-005938`（脚本历史默认前缀仍为 doubao），38 条事件，正常结束，无 `Traceback` / `ERROR` / `BadRequest` / `tool_choice` 写入日志。
- 回归测试：`uv run --group test pytest -o addopts='' -q` → `351 passed`；致命 Ruff 子集 `F821,F822,F823` 通过。

## 一、问题清单

### P1（严重）deepseek 的结构化决策被静默丢弃

**现象（pure_ds 纯 deepseek 局）**

- 女巫选了 `save`，但刀口仍死、**全局 0 个 `💊`**；
- 白天 5 人投票**仅 2 票生效**，多名 Agent 自述“我投了 2 号”却没计入；
- Agent 自我困惑：“我选了 save 但人没救 / 我投了 2 号但没计票”。

**受控复现**（`_play/repro_witch.py`，首夜有解药、刀口=3号）

| 模型                | `generate_response` 工具调用                                         | `invoke_structured`           | 女巫最终决策                   |
| ------------------- | -------------------------------------------------------------------- | ----------------------------- | ------------------------------ |
| deepseek-v4-flash   | **0 次**（决策写进 content 文本，如 `{"action":"save","seat":0,…}`） | `None` ×5                     | **save=0 / none=1 / poison=4** |
| doubao-seed-2.0-pro | 3 次（真实 `tool_use`）                                              | `WitchNightDecision(save)` ×3 | **save=3**                     |

**根因（两个缺陷叠加，与 max_tokens 无关）**

- **B1 — deepseek 不发工具调用**：`deepseek-v4-flash` 把结构化决策当**纯文本 JSON** 写进 `content`（它其实**正确选了 save**），不调用 AgentScope `ReActAgent` 依赖的 `generate_response` 工具。
    - `agentscope_agent.py::get_structured_response`（约 261–345）只从 `Msg.metadata` 取决策；metadata 为空时，对**非 Speech** 的 schema 直接 `return None`（约 321–329），content 里的真实 JSON 被丢弃。
    - `structured_invoke.py::invoke_structured`（约 48–102）只在**异常**时 `log.warning`；`get_structured_response` 返回 `None`（不抛异常）时**无任何日志** —— 这才是“静默”的真正出处（复现里 deepseek 的 structured 警告数 = 0）。
- **B2 — 夜间/投票的文本兜底是坏的**：`invoke_structured` 返回 None 后，`bridge.py::request_witch_night_choice`（约 393–436）/ `request_seat_choice`（约 578–656）回落到 `agent.get_response`；但这类 prompt 被 `agentscope_agent.py::_message_expects_seat_only` 判为“仅座位号”，`get_response` 在 agentscope 调用无有效输出时返回**写死的角色套话**（`_generate_role_specific_fallback`，如“我手里有药，但不会轻易使用……”）。bridge 再用 救/毒 关键字匹配套话 → 没“救”变 `none`，含“毒”变 `poison seat=0`（无效空毒）。**模型 content 里已给出的 save 全程无人解析。**

**为什么发言没事**：发言 prompt 不属于“仅座位号”，文本兜底返回模型真实发言文本，被 `WerewolfAdapterBridge.parse_speech` 正确解析。破坏只发生在「结构化决策 + 仅座位号 prompt」（女巫夜间、座位投票）。

**影响**：deepseek 女巫 `save` **5/5 全丢**；技能/投票被悄悄改写 → 阵营平衡与胜率失真、复盘/PostGame 数据被污染、且无日志难排查。doubao 不受影响。

### P2（性能）doubao 对局极慢

**现象**：单是 day-1 讨论就跑了约 20 分钟 / 3000+ 行；进不到投票。

**根因（两层叠加）**

- doubao 每次决策产出海量 `reasoning_content`，且 ReAct 循环常多走一轮“Now generate a text response”的自由发挥文本（1 次决策 ≈ 2 次模型调用量）。
- `information_hub.py::run_roundtable` 在**每条发言后**对**全体存活 Agent** 再采集一次投票意向（`_collect_vote_intentions`，约 206–264 / 313–321）→ 单个讨论轮的调用数约 **N² 量级**；叠加 `.env` 的 `AGENT_SERIAL_DELAY_SECONDS=1.0` 串行延迟，被进一步放大。

**影响**：纯性能（决策本身正确），但使 doubao 局几乎不可用于批量测试/评估。

---

### P3（轻微）女巫信息隔离：解药用完仍泄露刀口

**根因**：`roles/villager.py` 女巫 `get_private_notes`（约 95–103）**无条件**追加 `今晚狼人目标是X`，不判断 `has_save_potion`；而 `role_night_plans.py`（约 314–315）在解药用完时又声明“本夜不会告知刀口身份”。两层自相矛盾 → 解药用完仍能看到刀口（人机 doubao 局夜2 实测复现）。

**影响**：女巫获得不应有的信息，轻微破坏规则与信息隔离。

## 二、修复方案

### P1 修复（核心）

#### B1（主修）：在 `get_structured_response` 中解析 content-JSON

**目标**：模型把结构化决策写进 content（合法 JSON）时也能识别 —— 一处改动救回 deepseek 真实决策，**模型无关**。

新增工具函数（`agent_team/structured_invoke.py`）：

```python
def parse_structured_from_text(text: str, model: Type[T]) -> T | None:
    """从纯文本中尽力抽取并校验为结构化 model（容忍 markdown 围栏 / 前后散文 / tool-call 包装）。"""
    if not text or not text.strip():
        return None
    candidates: list[str] = [text.strip()]
    for obj in _iter_balanced_json_objects(text):  # 扫描所有平衡 {...}
        candidates.append(obj)
    last_valid: T | None = None
    for raw in candidates:
        try:
            data = json.loads(raw)
        except Exception:
            continue
        if isinstance(data, dict):
            # 解包 tool-call 风格：{"name":"generate_response","input":{...}} / {"arguments":{...}}
            if isinstance(data.get("input"), dict):
                data = data["input"]
            elif isinstance(data.get("arguments"), dict):
                data = data["arguments"]
        try:
            last_valid = model.model_validate(
                data
            )  # 取“最后一个能校验通过的对象”，避开 prompt 回显的示例 JSON
        except ValidationError:
            continue
    return last_valid
```

在 `agentscope_agent.py::get_structured_response` 的 `if text.strip():` 分支里，**非 Speech** schema 返回 None 之前先尝试：

```python
if text and text.strip():
    if structured_model is SpeechDecision:
        return WerewolfAdapterBridge.parse_speech(text)
    recovered = parse_structured_from_text(text, structured_model)  # ← 新增
    if recovered is not None:
        self.chat_history.append({"role": "assistant", "content": recovered.model_dump_json()})
        return recovered
return None
```

覆盖的 schema：`WitchNightDecision / SeatChoiceDecision / MultiSeatChoiceDecision / YesNoDecision / VoteIntentionDecision`。

#### B2（加固）：兜底解析原始文本 + 不再静默

1. **可观测**：`invoke_structured` 在最终返回 `None` 时补 `log.warning("structured_invoke_returned_none ...")`；`get_structured_response` 走到 `return None` 同样补 warning。让“决策被丢弃”可见。
2. **兜底走原始文本而非套话**：为 agent 增加 `get_raw_response(prompt)`（返回模型原始文本，**不**做 `_generate_fallback_response` 套话替换、**不**做 `[[N]]` 归约）。`request_witch_night_choice` / `request_seat_choice` 的无结构化兜底改用 `get_raw_response`，并对其先跑 `parse_structured_from_text`、再退化到现有 救/毒/座位号 关键字解析。这样即使 B1 漏网，兜底也解析真实文本而非编造的套话。
3. **（可选）让 deepseek 发工具调用**：`factory.py::create_react_agent` 对支持的端点设置 `tool_choice="required"` 或开启 JSON mode。有 B1 后此项非必需，列为增强项（且 vendor 相关、需灰度）。

> 取舍：B1 是“召回真实决策”，B2 是“防御 + 可观测”。建议二者一起做；若只做一项，先做 **B1**（收益最大、风险最低）。

---

### P2 修复：vote-intention 采样开关 + 限制 ReAct 文本回合

1. **vote-intention 可配置**（最大杠杆）：在 `PlayersConfig`/`GameConfig` 增加
    `vote_intention_mode: full | initial_only | sampled | off`（默认 `initial_only`）。
    `run_roundtable` 据此决定是否在每条发言后采集：
    - `full`：现状（每条发言后全体采集，N²）；
    - `initial_only`：仅讨论开始采一次（去掉 N×M）；
    - `sampled`：每 k 条发言采一次，或仅采子集观察者；
    - `off`：完全跳过。
        **取舍**：`PostGame` 的投票摇摆（vote-swing）分析依赖逐条意向，调低会降低该分析粒度——在文档与 CLI 帮助里写明。
2. **限制 ReAct 多余文本回合**：核对 `factory.py` 里 `ReActAgent` 的迭代配置（如 `max_iters`），对**决策类调用**限定单轮工具调用即结束，去掉决策后那轮“Now generate a text response”的自由发挥。
3. （可选）决策类 prompt 精简，减少 doubao 的 reasoning 体量。

---

### P3 修复：女巫私有提示按解药状态门控

`roles/villager.py` 女巫 `get_private_notes`：仅在仍持有解药时披露刀口，与 `role_night_plans.can_see_victim` 对齐：

```python
if (
    game_state and game_state.werewolf_target and self.has_save_potion
):  # ← 增加 has_save_potion 守卫
    target = game_state.get_player(game_state.werewolf_target)
    if target:
        notes.append(f"今晚狼人目标是{target.name}。")
```

## 五、风险与回滚

| 改动                             | 风险                             | 缓解 / 回滚                                                |
| -------------------------------- | -------------------------------- | ---------------------------------------------------------- |
| B1 解析 content-JSON             | 误解析 prompt 回显里的示例 JSON  | 取“最后一个能校验通过的对象”；schema 校验兜底；加单测      |
| B2 `get_raw_response` + 兜底改造 | 改动 bridge 决策路径，影响面较广 | 先跑 `gitnexus_impact`；保留旧关键字解析作为最后兜底；灰度 |
| P2 vote_intention 采样           | 降低 PostGame 摇摆分析粒度       | 配置项默认 `initial_only`，`full` 可随时恢复；文档写明取舍 |
| P3 门控                          | 极小                             | 单行守卫，独立提交                                         |

**优先级建议**：先合 **P1-B1**（最高收益）与 **P3**（最低风险），再做 **P1-B2** 与 **P2**。
