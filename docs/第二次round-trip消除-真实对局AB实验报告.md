# ③ 无用第二次 round-trip 消除 · 真实对局 A/B 实验报告

> 日期：2026-05-28
> 目标：用**真实 doubao 对局**验证"消除结构化决策后被丢弃的第二次 LLM 往返"（性能报告 ③）的实际效果——既要量化提速，也要确认不伤对局质量。
> 配套：实现见分支 `fix-second-round-trip`（`FastReActAgent` + `factory` 接线）；瓶颈定性见 [`对局性能瓶颈-影响程度量化报告.md`](%E5%AF%B9%E5%B1%80%E6%80%A7%E8%83%BD%E7%93%B6%E9%A2%88-%E5%BD%B1%E5%93%8D%E7%A8%8B%E5%BA%A6%E9%87%8F%E5%8C%96%E6%8A%A5%E5%91%8A.md)。

---

## 0 · TL;DR

- 🟦 **③ 的指纹被干净地消除**：未修复局有 **36.4%** 的 HTTP 往返是"只吐文本、无工具调用"的浪费第二次往返（187 次中 68 次）；修复后该比例 **= 0%**（55 次全部携带工具调用）。
- 🟦 **浪费的 reasoning token 归零**：未修复 **38,518** → 修复后 **0**——doubao 正是在那次被丢弃的第二次往返里产出独白式 reasoning，修复后这部分彻底消失。
- 🟩 **确定性证据**：单元测试 `test_react_single_roundtrip.py` 证明"模型只吐工具调用、不吐文本"时，每个结构化决策的模型调用次数 **2 → 1**。
- 🟢 **对局质量无回退**：质量评审对比两臂 transcript，结论 `NO_QUALITY_REGRESSION`——发言完整流畅（均长 75.5 → 70.2 字）、零空发言/占位符、夜间与投票决策合理、推理连贯。
- ⚠️ **诚实口径**：两局因 LLM 非确定性走出了**不同长度**（未修复 2 个游戏轮、村民胜；修复 1 个游戏轮、狼人胜），故**原始 4.4× 墙钟差不能直接归因于本修复**。可归因于 ③ 的、与局长无关的提速估计为**墙钟的 ~35–43%**（见 §3）。

---

## 1 · 方法与口径

**两臂**（同 seed=12345、同 6p doubao 配置 `llm-6p-doubao.yaml`、意向轮询保持默认开启）：

| 臂         | 代码状态                     | ReActAgent        | 期望 ③ 行为                    |
| ---------- | ---------------------------- | ----------------- | ------------------------------ |
| **before** | `main` 源码（未修复）        | 原版 `ReActAgent` | 结构化决策后多打一次纯文本往返 |
| **after**  | 分支 `fix-second-round-trip` | `FastReActAgent`  | 首轮即终止，无第二次往返       |

**计时器**（`scripts/timed_game.py`，自包含、非侵入）：只在 `openai` 边界 patch `AsyncCompletions.create`，记录每次**物理 HTTP 往返**的 wall / tokens / finish_reason / tool_calls；同时落一份事件流 transcript 供质量评审。补丁内只累加不 print（避开 rich/logfire 的 stdout 捕获陷阱）。两臂用**同一份脚本**、仅切 `PYTHONPATH` 指向不同源码树，确保指标口径一致。

**证据等级**：🟩 确定性（代码/单测）｜🟦 实测（本次真实对局插桩）｜🟨 估算（由实测推导，含假设）。

**关键混淆项（必须正视）**：两局是独立的真实对局，LLM 温度导致剧情分叉——未修复局打满 2 个游戏轮（村民胜），修复局 1 个游戏轮即结束（狼人胜）。**对局长度不同 ⇒ 原始总量（总墙钟、总 HTTP、总 token）不可直接对比**。因此本报告以\*\*与局长无关的"浪费占比"\*\*作为承重证据，原始总量只作旁证。

## 2 · 实测结果

| 指标                     | before（未修复）     | after（修复后）      | 说明                         |
| ------------------------ | -------------------- | -------------------- | ---------------------------- |
| 总墙钟                   | 3031.6 s（50.5 min） | 694.2 s（11.6 min）  | ⚠️ 含局长差，不可直接归因    |
| 游戏轮数                 | 2 轮（+ setup）      | 1 轮（+ setup）      | 剧情分叉，非修复所致         |
| 胜负                     | 村民胜               | 狼人胜               | 非确定性，非质量信号         |
| **HTTP 往返总数**        | **187**              | **55**               |                              |
| ├ 带工具调用             | 119                  | 55                   | 真正承载决策的调用           |
| └ **纯文本无工具调用**   | **68（36.4%）**      | **0（0%）**          | 🟦 **③ 的直接指纹**          |
| **reasoning token 合计** | **38,518**           | **0**                | 🟦 第二次往返的浪费独白      |
| HTTP 占墙钟比            | 96.4%                | 91.9%                | 墙钟几乎全在等模型           |
| 单次 HTTP 均值 / 中位    | 15.62 s / 11.57 s    | 11.60 s / **5.93 s** | 🟦 中位大幅下降              |
| finish_reason=length     | 15                   | 7                    | ⑤ 问题两臂都在，非本修复目标 |
| prompt token 合计        | 3,808,363            | 732,909              | ⚠️ 受局长影响，见 §3         |

**承重结论（🟦，与局长无关）**：未修复局每 ~3 次 HTTP 就有超过 1 次是纯浪费的第二次往返（36.4%）；修复后这类调用**彻底消失**（0%），且与之绑定的 reasoning token 从 38,518 归零。这与确定性单测（§4）完全自洽。

## 3 · 时间收益的诚实估计

**不能说"快了 4.4×"**——那主要来自修复局少打了一个游戏轮。可归因于 ③ 的提速，应在"同等长度对局"口径下估计：

🟨 **臂内估算（only ③，与局长无关）**：未修复局 68 次浪费往返。

- 若按全局均值 15.62 s/次：68 × 15.62 ≈ **1062 s**，占该局墙钟 **~35%**。
- 浪费往返是 reasoning-heavy 的（diagnosis 探针实测第二次往返 ~19 s，高于首次 ~8 s）；若按 ~19 s/次：68 × 19 ≈ **1292 s**，占 **~43%**。
- ⇒ **③ 单独可削减同等对局约 35–43% 的墙钟**。HTTP 占墙钟 96%，故"减往返"几乎等比转化为减墙钟。

🟨 **每轮归一（次要，仍有噪声）**：before 1010.5 s/轮、62.3 HTTP/轮；after 347.1 s/轮、27.5 HTTP/轮。每轮墙钟 ~2.9×、每轮 HTTP ~2.3×。但该口径仍被污染：未修复局首轮 6 人存活时 ② 的 O(N²) 意向轮询最猛（单轮 2150 s），与修复无关——故每轮差里混入了 ②×存活人数效应，不能全算到 ③ 头上。

🟨 **token**：prompt token 3.8M → 0.73M（5.2×）。其中一部分是 ③ 收益（每次被消除的第二次往返本会重发整段增长的记忆），但大部分受局长（轮数越多累计上下文越大）影响，**无法干净拆分**，仅作旁证。

> 一句话：**承重的是"36.4% → 0% 浪费往返"与确定性 2→1**；墙钟收益诚实区间是**同等对局 ~⅓–⅖**，而非观测到的 4.4×。

## 4 · 确定性证据（CI 级，无需 API）

`tests/agent_team/test_react_single_roundtrip.py`（真实 `FastReActAgent` + 计数假模型）：

- **T1 往返计数**：模型只吐 `generate_response` 工具调用、不吐文本时，模型调用次数 == 1（修复前为 2，RED 失败 `assert 2 == 1`）。
- **T2 对照**：模型同步吐工具调用 + 文本时仍 == 1（不破坏既有快路径）。
- **T3 prompt 合法性**：补的空 `TextBlock` 只作用于返回消息、**不写入 memory**；每个 tool_call 仍被 tool_result 应答，末尾无残留纯文本轮。

全量 `tests/agent_team` 85 passed（基线 84 + 新增 1），零回归；计时器单测 2 passed。

## 5 · 对局质量评审

独立质量评审 agent 通读两臂 transcript，逐维度对比，结论 **`NO_QUALITY_REGRESSION`**：

- **发言质量**：两臂发言均完整、切题、中文流畅。长度分布几乎一致——before 13 条均 75.5 字（51–98），after 11 条均 70.2 字（47–87）；**两臂均零空发言 / 零占位符 / 无截断**。这关掉了本修复唯一的理论风险（返回消息文本变空是否殃及 `public_speech`）：发言始终由结构化 `data.speech` 承载，after 完整保留。
- **决策合理性**：两臂夜间技能（预言家查验、女巫救/毒、狼刀）与投票均结构良好、reason 非空、`fallback: false`。after 中女巫首夜选择不救属标准打法；平票触发 PK 轮被正确处理。
- **推理连贯性**：after 发言正确引用真实局面（如"4 号夜里死了大概率是神职"对应预言家被刀），无推理质量下降。

**底线**：修复后对局质量**不低于**修复前；本修复只移除了无人读取的第二次自由文本，结构化决策与发言文本完好。

## 6 · 不确定性与局限

- 🟦 **单样本**：每臂仅 1 局，未做多局方差。墙钟与单次延迟受网络/模型负载波动影响；承重指标（浪费往返占比、reasoning token、单测 2→1）不受此影响。
- ⚠️ **局长分叉**：两局轮数/胜负不同，已在 §3 显式扣除，原始总量仅旁证。
- **②、⑤ 仍在**：意向轮询（②，两臂都开）与 `max_tokens=2048`/无超时（⑤，两臂 finish_length 15/7）均未被本修复触及——它们是后续独立杠杆。本实验只隔离 ③。
- **模型相关**：结论基于 doubao（必中第二次往返分支）。deepseek 是否中招取决于其是否顺带吐文本，需单独测。

**消除不确定性的下一步**：多局重复（≥3）取均值刷新 §3 估算；可选地在计时器里按 tool/no-tool 分桶记录每次 wall，把 §3 的 🟨 估算升级为 🟦 实测。
